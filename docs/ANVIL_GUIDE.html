Q — Quantity

A number with a physical unit. Arithmetic propagates dimensions automatically. Incompatible units raise errors at computation time.

⚙️

Relation (RSQ)

A plain Python function that returns a dict. Inputs detected from signature, outputs from return keys. Registered in SQLite with metadata.

🔗

System

A computation graph — add named quantities, wire in relations, then solve forward, sweep parameters, or optimize. Returns a Result object.

Why Anvil

Engineering computation sits at an awkward intersection: the mathematics is well-understood, the physics is domain-specific, and the tooling is fragmented. Physicists reach for numpy and scipy. Chemical engineers reach for Aspen. Aerospace engineers reach for MATLAB. Nobody agrees on how to share physics knowledge across projects, or even across the same project over time.

Anvil is built around a specific observation: the primary unit of reusable engineering knowledge is a pure function that maps physical inputs to physical outputs. Not a class, not a module, not a simulation — a function. Once you accept that, the rest follows.

The core insight: If a physics computation is a plain Python function (kwargs → dict), it can be stored, searched, version-controlled, composed into systems, swept over parameter spaces, and called from any domain — without any framework coupling. The function doesn't need to know it's in Anvil. Anvil just provides the infrastructure around it.

What the Existing Tools Get Wrong

The tools Anvil is most often compared to each solve part of the problem but leave a gap.

Pint / astropy.units

Unit tracking only. No computation graph, no registry, no solvers, no sweep. You still assemble everything by hand in scripts. Units are annotations — not participants in a system.

OpenMDAO

Computation graphs with derivatives. But: components require subclassing a framework class, explicit connection declarations in XML/YAML, no unit-first arithmetic, no knowledge registry. Optimized for gradient-based MDO, not general engineering scripting.

scipy / sympy

Excellent solvers and symbolic math. But stateless — no concept of a persistent workspace, no unit propagation, no reusable function registry. Every analysis starts from scratch.

CoolProp / REFPROP

Best-in-class fluid property databases. But isolated — no graph, no units, no composition. Using CoolProp in an engineering system still requires manual wiring and unit conversion.

MATLAB / Simulink

Time-domain simulation and control design. No persistent physics registry, no unit-first arithmetic, closed ecosystem, commercial license. Sharing a function between projects means copying files.

Excel / spreadsheets

Still the most common "engineering computation" tool. Zero composability, no units, no version control, no search, no testing. Every spreadsheet reinvents the same formulas.

What Anvil Does That Nothing Else Does

1. Graph wiring from function signatures. When you call sys.use(my_fn), Anvil reads the argument names of my_fn and connects them to matching quantities already in the workspace. You never declare connections explicitly. The function signature is the connection diagram. This is the most direct expression of the physical structure: if a function needs rho and V, it says so in its definition — no separate XML, no decorator, no class attribute.

2. Source-level function persistence. The registry stores the full Python source text of every RSQ. This means functions survive Python restarts, can be retrieved on any machine with Anvil installed, can be exported to .py files, searched by keyword, and versioned. No other Python engineering tool has a persistent, searchable, source-level function store. The closest analogy is a paper's appendix — a permanent, citable record of the method.

3. Units that participate in computation, not just annotate it. In Pint, you annotate an existing computation with units. In Anvil, the computation is between quantities — the units are there from the start, propagate through every operation, and dimension errors surface at the point of incompatibility. A Q divided by another Q knows its resulting dimension without being told.

4. The RSQ as the atomic unit of engineering knowledge. Most frameworks operate at the level of modules or classes. Anvil operates at the level of individual physics functions. This granularity matters: a single RSQ can be tested in isolation, pushed to the registry independently, composed with unrelated RSQs, and called directly without loading a system. It is the minimal viable unit of reusable physics.

When Not to Use Anvil

Anvil is not the right tool for every problem. Be specific about where it fits:

  • Large-scale FEM / CFD — Anvil includes a 2D structured Euler solver (anvil.cfd) for parametric studies. For viscous, turbulent, unstructured, or 3D flows use OpenFOAM/SU2/ANSYS via a CLI Adapter.
  • Symbolic algebra — Use SymPy. Anvil operates numerically.
  • Real-time control loops — Anvil solvers are not designed for sub-millisecond execution.
  • Gradient-based MDO at scale — OpenMDAO with analytic derivatives is faster for large design spaces requiring exact Jacobians.
  • EOS / thermophysical property accuracy — CoolProp and REFPROP are more accurate for near-critical and multi-phase conditions. Use them via Anvil's adapter pattern.

Anvil is the right tool when you are: writing engineering physics in Python, want to reuse functions across projects, need unit safety without ceremony, and want sweep/sensitivity as first-class operations — not as afterthoughts assembled from scipy primitives every time.

Arithmetic — Unit Propagation

# Arithmetic propagates dimensions automatically
F   = Q(100, "N")
A   = Q(0.01, "m^2")
sig = F / A           # Q(10000, "Pa")  ← dimension inferred as [L⁻¹M T⁻²]

KE  = 0.5 * Q(10, "kg") * Q(30, "m/s")**2  # Q(4500, "J")

print(sig)   # 10000.00 Pa
print(KE)    # 4500.00 J
print(Q(1, "km") + Q(500, "m"))   # 1.5000 km  ← unit hint from left operand
print(Q(100, "m") * 2)              # 200.00 m
print(9.81 * Q(1, "kg"))            # 9.8100 kg  ← dimensionless × [M] = [M]
python
Output
10000.00 Pa 4500.00 J 1.5000 km 200.00 m 9.8100 kg
What changes this:
  • Multiply/divide → exponents add/subtract. N / m² = Pa, kg × (m/s)² = J.
  • Add/subtract → dimensions must match exactly. Mixed units (km + m) convert to SI before adding.
  • Scalar × Q → unit unchanged. Anvil cross-references the resulting dimension against the database to pick a named unit.
Tip — .si extraction: When passing Q values to external functions that expect plain floats, use .si to get the SI float. Use .value to get the display-unit float.

Relation (RSQ)

A Relation is any Python function that: accepts keyword arguments, returns a dict of outputs. Inputs are auto-detected from the function signature; outputs are the dict keys.

# Minimal Relation — plain function (most common)
def ideal_gas_density(P, T, R_gas=287.0):
    """Air density from ideal gas law. P [Pa], T [K], R_gas [J/kg/K]."""
    from anvil import Q
    return {"rho": Q(P / (R_gas * T), "kg/m^3")}

# Direct call
r = ideal_gas_density(P=101325, T=300)
print(r["rho"])   # 1.1763 kg/m^3
python
Output
1.1763 kg/m^3
What changes this: P ÷ (R_gas × T) = density. Higher P → higher density. Higher T → lower density. Default R_gas=287 is for dry air — change it for other gases.

from anvil.relation import Relation

rel.info()    # shows inputs, outputs, defaults

Multi-step Block

# Chain multiple functions into one Relation (outputs feed next step)
def step1(M, gamma=1.4):
    T_ratio = 1 + (gamma-1)/2 * M**2
    return {"T_ratio": T_ratio, "gamma": gamma}

def step2(T_ratio, gamma):        # uses T_ratio from step1
    P_ratio = T_ratio**(gamma/(gamma-1))
    return {"P_ratio": P_ratio}

isentropic = Relation.block("isentropic", steps=[step1, step2],
                             tags=["compressible"], desc="Isentropic ratios")
isentropic(M=2.0)  # {"T_ratio": 1.8, "gamma": 1.4, "P_ratio": 7.824}
python
Output
{'T_ratio': 1.8, 'gamma': 1.4, 'P_ratio': 7.824}
What changes this: M (Mach number) drives both. At M=1: T_ratio=1.2, P_ratio=1.893. At M=2: T_ratio=1.8, P_ratio=7.824. At M=3: T_ratio=2.8, P_ratio=36.73. Steeper pressure rise than temperature rise because the exponent is γ/(γ-1)=3.5 for air.

Rules for RSQ Functions

  • Accept inputs as keyword arguments only
  • Return a dict — keys become output names
  • Inputs with defaults are optional; Anvil won't require them from the workspace
  • For registry storage: imports must be inside the function body (lazy import)
  • Output values can be plain floats, Q objects, arrays, or booleans

System

A System is a named computation graph. Add named quantities, wire in Relations, then solve. Anvil determines topological order automatically.

import anvil
from anvil import Q, System

nozzle = System("rocket_nozzle")

# Add quantities: name, value, unit (optional), desc (optional)
nozzle.add("P0",      10e6,  "Pa",    desc="Chamber pressure")
nozzle.add("T0",      3500,  "K",     desc="Chamber temperature")
nozzle.add("gamma",   1.2)
nozzle.add("A_throat",0.005, "m^2")
nozzle.add("A_exit",  0.08,  "m^2")
nozzle.add("P_amb",   0,     "Pa")    # vacuum

# Wire in Relations (by object, string name, or inline function)
nozzle.use("nozzle_area_ratio")          # from global registry
nozzle.use("area_mach_supersonic")
nozzle.use("isentropic_ratios")
nozzle.use(my_thrust_fn)                  # inline function

# Update a quantity value
nozzle.set(T0=3200)   # keyword style

Optimization

# Maximize / minimize an output by varying design variables
opt = sys.optimize(
    objective="Isp",           # output key to optimize
    design_vars={
        "P0":    (5e6,  30e6),  # (lo, hi) in declared units
        "T0":    (2500, 4000),

    minimize=False,             # True = minimize, False = maximize

print(opt.x)          # dict of optimal design variable values

Registry

The registry is the backbone of Anvil's reusability story. It is a SQLite database stored at ~/.anvil/registry.db. Every RSQ (Relation) you push is serialized as its full Python source text and stored with metadata — domain, tags, description, version. On any machine with Anvil installed, you can retrieve and run that RSQ by name without re-defining it.

anvil.push(fn) ──────────────────────────── ▶ ~/.anvil/registry.db │ stores: { name, source_code, domain, │ SQLite table: rsqs tags, description, version } │ │ anvil.R.my_rsq(...) ◀──── deserialize + exec source ◀──┘ lazy: compiled on first call

Self-containment Rule

All imports must be inside the function body. The registry stores only the function's source code — the surrounding module namespace is not captured. When the RSQ is later deserialized and called, np, scipy, and Q are not in scope unless you import them inside the function. This is the most common source of registry bugs.
# CORRECT — imports inside body
def bernoulli(P1, v1, rho, P2):
    import numpy as np
    from anvil import Q
    v2 = np.sqrt(2*(P1 - P2)/rho + v1**2)
    return {"v2": Q(v2, "m/s")}

# WRONG — top-level import won't survive serialization
import numpy as np          # ← not stored in registry
def bernoulli_bad(P1, v1, rho, P2):
    v2 = np.sqrt(...)         # ← NameError when loaded from registry
    return {"v2": v2}
python

How Auto-Seeding Works

On the very first import anvil, Anvil calls seed() which inserts all 76 built-in RSQs into the global registry. On subsequent imports, it compares source strings — if any built-in has changed (e.g., after an Anvil upgrade), only the changed RSQs are re-seeded. Your own pushed RSQs are never touched by seeding.

Seven RSQs covering the core signal-processing toolkit: FFT spectrum, STFT spectrogram, Welch PSD, Butterworth filtering, Hilbert envelope, cross-correlation, and descriptive statistics. All accept plain NumPy arrays as inputs and return arrays + scalar summaries. Domain: misc.

fft_spectrum — one-sided power spectrum

import numpy as np, anvil

t  = np.linspace(0, 1, 1024, endpoint=False)
dt = t[1] - t[0]
sig = np.sin(2*np.pi*50*t) + 0.3*np.sin(2*np.pi*150*t)  # 50 Hz + 3rd harmonic

r = anvil.R.fft_spectrum(signal=sig, dt=dt, window="hann")
print("dominant_freq:", round(r["dominant_freq"], 1))
print("RMS:          ", round(r["rms"], 4))
print("THD:          ", round(r["thd"], 4))
print("f_resolution: ", round(r["f_resolution"], 3), "Hz")
python
Output
dominant_freq: 50.0 RMS: 0.7399 THD: 0.2999 f_resolution: 0.977 Hz
What changes this: window="hann" reduces spectral leakage for non-integer-cycle signals. window="none" uses a rectangular window — fine for exactly periodic signals, leaky otherwise. THD is √(sum of 2nd–5th harmonic powers) / fundamental amplitude — the 0.3 amplitude harmonic gives THD≈0.30. Longer signals → finer frequency resolution (1/T Hz).

Adapters

Adapters wrap external Python libraries or CLI tools as native Anvil Relations. Once wrapped, they plug into Systems, sweeps, and sensitivity analysis identically to plain functions.

AI-friendly by design. The adapter pattern is fully specified in this guide. Any LLM or AI coding agent can write a complete, working adapter for any package given two inputs: (1) this adapter guide section, and (2) the target package's documentation. The pattern is unambiguous — input/output spec, backend type, unit declarations, mock fallback — so the agent has no design decisions to make. Paste this section + the package docs into your agent and ask it to write the adapter.
from anvil import Adapter, Q

def _call(P, T):
    try:
        import CoolProp.CoolProp as CP
    except ImportError:
        raise ImportError("Install: pip install CoolProp")
    return {
        "rho": Q(CP.PropsSI('D', 'P', P, 'T', T, 'Water'), "kg/m^3"),
        "mu":  Q(CP.PropsSI('V', 'P', P, 'T', T, 'Water'), "Pa*s"),
    }

water = Adapter("coolprop_water",
    backend="python", call=_call,
    inputs={
        "P": {"unit": "Pa", "desc": "Pressure", "default": 101325},
        "T": {"unit": "K",  "desc": "Temperature", "default": 300},
    },
    outputs={
        "rho": {"unit": "kg/m^3", "desc": "Density"},
        "mu":  {"unit": "Pa*s",   "desc": "Dynamic viscosity"},
    },
    desc="Water properties via CoolProp",
    tags=["fluid", "water", "coolprop"],
)

# Direct call
r = water(P=101325, T=300)   # rho: 998.2 kg/m^3

# In a System — pass the Adapter object directly (NOT anvil.R.name)
sys.use(water)
python

Solvers

Standalone solvers in anvil.solvers, also available as System methods.

from anvil import solvers
import numpy as np

# Root finding — returns float directly
root = solvers.find_root(lambda x: x**3 - x - 1, x0=1.5)
print("root:", round(root, 4))

# Nonlinear system: x²+y²=1, x=y² — returns array
def equations(vars):
    x, y = vars
    return [x**2 + y**2 - 1, x - y**2]
sol = solvers.solve_nonlinear(equations, x0=[0.7, 0.8])
print("x,y:", sol.round(4))

# ODE IVP — free fall from 1000 m, returns dict with "t" and "y"
def free_fall(t, y):
    h, v = y
    return [v, -9.81]
sol = solvers.solve_ode(free_fall, y0=[1000, 0],
                          t_span=(0, 14), t_eval=np.linspace(0, 14, 5))
print("h(t):", sol["y"][0].round(1))

# Global optimization — returns dict with "x", "fun", "success"
sol = solvers.minimize_global(lambda x: (x[0]-2)**2 + (x[1]-3)**2,
                                bounds=[(-5,5),(-5,5)],
                                method="differential_evolution", seed=0)
print("min at:", [round(v,4) for v in sol["x"]], "  fun≈0")

Watchdog — Custom Convergence Checker

Watchdog is the convergence utility used internally by all iterative solvers. Use it directly when building custom iterative algorithms outside of a System.

from anvil.watchdog import Watchdog

wd = Watchdog(rtol=1e-6, max_iter=500)

current = initial_guess.copy()
for i in range(wd.max_iter):
    prev = current.copy()
    current = update_step(current)          # your iteration logic
    converged, residual = wd.check(prev, current)
    if converged:
        print(f"Converged in {i+1} iterations. Residual: {residual:.3e}")
        break
python
What wd.check(prev, current) computes: max relative change across all values: max(|curr - prev| / max(|prev|, ε)). Returns (True, residual) when below rtol. Use directly in custom fixed-point loops, custom Newton-Raphson variants, or any iterative scheme where you need consistent convergence criteria.

CFD Solver — 2D Euler

anvil.cfd is a native finite-volume solver for the 2D Euler (inviscid) equations on structured body-fitted meshes. Handles subsonic through supersonic flows including shocks. Integrates with Anvil Systems via solver.as_relation().

from anvil.cfd import CFDSolver, Mesh
from anvil.cfd.bc import SupersonicInlet, SupersonicOutlet, SlipWall, Farfield

# 1. Mesh — built-in geometry factories
mesh = Mesh.wedge(half_angle_deg=10, chord=1.0, height=0.8, nx=80, ny=40)
# also: Mesh.bump(), Mesh.cartesian(), Mesh.compression_ramp(), Mesh.from_arrays(X,Y)

# 2. Boundary conditions
M_inf, p_inf, T_inf = 2.0, 101325.0, 300.0
bcs = {
    "west":  SupersonicInlet(M=M_inf, p=p_inf, T=T_inf),
    "east":  SupersonicOutlet(),
    "south": SlipWall(),
    "north": Farfield(M=M_inf, p=p_inf, T=T_inf),
}

# 3. Solver
solver = CFDSolver(mesh, bcs, gamma=1.4, flux_scheme="roe", order=2, cfl=0.5)
solver.initialize(M=M_inf, p=p_inf, T=T_inf, alpha_deg=0.0)

# 4. Run
result = solver.run(max_iter=5000, tol=1e-6, monitor=True, print_every=200)
result.summary()
result.to_vtk("wedge.vtk")      # open in ParaView
result.to_tecplot("wedge.dat")  # open in Tecplot

Common Patterns

Pattern 1 — Build a System from Scratch

import anvil
from anvil import Q, System

# 1. Define a custom Relation
def reynolds_number(rho, V, L_char, mu):
    from anvil import Q
    return {"Re": Q(rho * V * L_char / mu, "1")}

# 2. Create System and add quantities
pipe = System("pipe_flow")
pipe.add("rho",    998.0,  "kg/m^3",  desc="water density")
pipe.add("V",      2.0,    "m/s")
pipe.add("D",      0.05,   "m",        desc="pipe diameter")
pipe.add("mu",     1.002e-3, "Pa*s",   desc="water viscosity at 20°C")

# 3. Wire in Relation (map "L_char" → "D")
pipe.use(reynolds_number, map={"L_char": "D"})

# 4. Solve and inspect
result = pipe.solve_forward()
result.summary()
python

What changes this: Re = ρ·V·D/μ. Re > 4000 → turbulent (this case: fully turbulent). Double V → double Re. Halve D → halve Re. Use anvil.db.fluids.get("water", T=Q(20,"degC")) to look up ρ and μ from the built-in database.

Pattern 2 — Chaining RSQs (output feeds next)

# RSQ outputs become available to downstream RSQs automatically
nozzle = System("nozzle")
nozzle.add("M_exit", 3.0)
nozzle.add("gamma",  1.4)
nozzle.add("P0",     5e6,  "Pa")
nozzle.add("T0",     3000, "K")

# map: RSQ input "M" reads workspace variable "M_exit"
nozzle.use("isentropic_ratios", map={"M": "M_exit"})  # → T0_T, P0_P, rho0_rho

# Inline lambdas use T0_T / P0_P from previous step
nozzle.use(lambda T0, T0_T: {"T_exit": Q(T0 / T0_T, "K")})
nozzle.use(lambda P0, P0_P: {"P_exit": Q(P0 / P0_P, "Pa")})

result = nozzle.solve_forward()
result.summary()
