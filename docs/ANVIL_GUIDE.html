Anvil Framework v1.3

Engineering computation — write physics as Python, get results with automatic unit tracking.

Anvil turns plain Python functions into solvable engineering systems. Declare physical quantities with units, wire them into computation graphs, and let Anvil handle solves, sweeps, and sensitivity — all with dimensional analysis at every step.

Core Concepts

Anvil has four building blocks that compose naturally:

┌──────────────────────────────────┐ │ Q (Quantity) │ number + unit + dimension │ Q(300, "K") → "300.00 K" │ arithmetic propagates units └──────────────────────────────────┘ │ inputs / outputs ▼ ┌──────────────────────────────────┐ │ Relation (RSQ) │ function: kwargs → dict │ def ideal_gas(P, V, T): │ auto-detects inputs & outputs │ return {"n": P*V/(R*T)} │ stored in SQLite registry └──────────────────────────────────┘ │ sys.use(relation) ▼ ┌──────────────────────────────────┐ │ System │ DAG of Quantities + Relations │ sys.add("P", 101325, "Pa") │ forward / Newton / sweep / optimize │ sys.solve_forward() │ returns Result with all values └──────────────────────────────────┘ │ anvil.push(fn, domain="aero") ▼ ┌──────────────────────────────────┐ │ Registry │ SQLite store, namespaced access │ anvil.R.isentropic_ratios(M=2) │ 76 built-in RSQs auto-seeded │ anvil.search("compressible") │ project-local registries too └──────────────────────────────────┘
📐

Q — Quantity

A number with a physical unit. Arithmetic propagates dimensions automatically. Incompatible units raise errors at computation time.

⚙️

Relation (RSQ)

A plain Python function that returns a dict. Inputs detected from signature, outputs from return keys. Registered in SQLite with metadata.

🔗

System

A computation graph — add named quantities, wire in relations, then solve forward, sweep parameters, or optimize. Returns a Result object.

Quantity — Q

Every physical value in Anvil is a Q object: a float paired with a unit string and a Dim dimension object.

from anvil import Q, K, Pa, m, s, kg, N, J, W, kPa, MPa, kJ

# Three equivalent ways to create
T  = Q(300, "K")       # explicit string unit
T  = Q(300, "temperature")  # category alias
T  = 300 * K            # UnitStub syntax (cleanest)

P  = 6.9 * MPa
rho = Q(1.225, "kg/m^3")
v   = Q(100, "m/s")

# Display: value  unit  (name if set)
print(T)     # 300.00 K
print(P)     # 6.9000 MPa
print(rho)   # 1.2250 kg/m^3
python
Output
300.00 K 6.9000 MPa 1.2250 kg/m^3
What changes this: The unit string in Q(value, "unit") sets the display. Internally, all values are stored in SI (Kelvin, Pascals, kg/m³). The printed format is always value unit with 4 decimal places.

Units & Conversion

# Convert to any compatible unit
print(Q(300, "K").to("R"))         # 540.00 R  (Rankine)
print(Q(101325, "Pa").to("bar"))   # 1.0132 bar
print(Q(1, "kg").to("lb"))          # 2.2046 lb
print(Q(100, "m/s").to("km/hr"))   # 360.00 km/hr

# Temperature with offset units (°C/°F)
print(Q(25, "degC"))               # 25.0000 degC  (stored as 298.15 K internally)
print(Q(25, "degC").to("K"))       # 298.15 K
print(Q(25, "degC").to("degF"))    # 77.00 degF
print(Q(100, "degC").to("degF"))  # 212.00 degF
print(Q(32, "degF").to("K"))       # 273.15 K

# Angles — .si is always radians
print(Q(90, "deg").si)              # 1.5708  (π/2 radians)
print(Q(90, "deg").to("rad"))      # 1.5708 rad
print(Q(3.14159, "rad").to("deg")) # 180.00 deg

# Incompatible conversion raises ValueError
Q(300, "K").to("m")           # ValueError: [TH] to [L] incompatible
python
Output
540.00 R 1.0132 bar 2.2046 lb 360.00 km/hr 25.0000 degC 298.15 K 77.00 degF 212.00 degF 273.15 K 1.5708 1.5708 rad 180.00 deg
What changes this: .to("unit") never alters the internal SI value — it creates a new Q with a different display hint. For °C/°F, Anvil applies the offset formula automatically: K = °C + 273.15, K = (°F + 459.67) × 5/9.
Important — deg/rad in RSQs: All built-in RSQs with _deg parameters (e.g. i_deg, RAAN_deg) accept either a plain float (treated as degrees) or a Q(value, "deg") object. Both give identical results. This means you can store angles in a System as Q("deg") and RSQs will convert correctly.

Arithmetic — Unit Propagation

# Arithmetic propagates dimensions automatically
F   = Q(100, "N")
A   = Q(0.01, "m^2")
sig = F / A           # Q(10000, "Pa")  ← dimension inferred as [L⁻¹M T⁻²]

KE  = 0.5 * Q(10, "kg") * Q(30, "m/s")**2  # Q(4500, "J")

print(sig)   # 10000.00 Pa
print(KE)    # 4500.00 J
print(Q(1, "km") + Q(500, "m"))   # 1.5000 km  ← unit hint from left operand
print(Q(100, "m") * 2)              # 200.00 m
print(9.81 * Q(1, "kg"))            # 9.8100 kg  ← dimensionless × [M] = [M]
python
Output
10000.00 Pa 4500.00 J 1.5000 km 200.00 m 9.8100 kg
What changes this:
  • Multiply/divide → exponents add/subtract. N / m² = Pa, kg × (m/s)² = J.
  • Add/subtract → dimensions must match exactly. Mixed units (km + m) convert to SI before adding.
  • Scalar × Q → unit unchanged. Anvil cross-references the resulting dimension against the database to pick a named unit.
Tip — .si extraction: When passing Q values to external functions that expect plain floats, use .si to get the SI float. Use .value to get the display-unit float.

Relation (RSQ)

A Relation is any Python function that: accepts keyword arguments, returns a dict of outputs. Inputs are auto-detected from the function signature; outputs are the dict keys.

# Minimal Relation — plain function (most common)
def ideal_gas_density(P, T, R_gas=287.0):
    """Air density from ideal gas law. P [Pa], T [K], R_gas [J/kg/K]."""
    from anvil import Q
    return {"rho": Q(P / (R_gas * T), "kg/m^3")}

# Direct call
r = ideal_gas_density(P=101325, T=300)
print(r["rho"])   # 1.1763 kg/m^3
python
Output
1.1763 kg/m^3
What changes this: P ÷ (R_gas × T) = density. Higher P → higher density. Higher T → lower density. Default R_gas=287 is for dry air — change it for other gases.
# Wrap with metadata
from anvil.relation import Relation
rel = Relation(ideal_gas_density, tags=["thermo", "air"],
               desc="Air density via ideal gas law")
rel.info()    # shows inputs, outputs, defaults
python
Output
ideal_gas_density inputs : P, T defaults: R_gas=287.0 outputs : rho tags : ['thermo', 'air'] desc : Air density via ideal gas law

Multi-step Block

# Chain multiple functions into one Relation (outputs feed next step)
def step1(M, gamma=1.4):
    T_ratio = 1 + (gamma-1)/2 * M**2
    return {"T_ratio": T_ratio, "gamma": gamma}

def step2(T_ratio, gamma):        # uses T_ratio from step1
    P_ratio = T_ratio**(gamma/(gamma-1))
    return {"P_ratio": P_ratio}

isentropic = Relation.block("isentropic", steps=[step1, step2],
                             tags=["compressible"], desc="Isentropic ratios")
isentropic(M=2.0)  # {"T_ratio": 1.8, "gamma": 1.4, "P_ratio": 7.824}
python
Output
{'T_ratio': 1.8, 'gamma': 1.4, 'P_ratio': 7.824}
What changes this: M (Mach number) drives both. At M=1: T_ratio=1.2, P_ratio=1.893. At M=2: T_ratio=1.8, P_ratio=7.824. At M=3: T_ratio=2.8, P_ratio=36.73. Steeper pressure rise than temperature rise because the exponent is γ/(γ-1)=3.5 for air.

Rules for RSQ Functions

  • Accept inputs as keyword arguments only
  • Return a dict — keys become output names
  • Inputs with defaults are optional; Anvil won't require them from the workspace
  • For registry storage: imports must be inside the function body (lazy import)
  • Output values can be plain floats, Q objects, arrays, or booleans

System

A System is a named computation graph. Add named quantities, wire in Relations, then solve. Anvil determines topological order automatically.

import anvil
from anvil import Q, System

nozzle = System("rocket_nozzle")

# Add quantities: name, value, unit (optional), desc (optional)
nozzle.add("P0",      10e6,  "Pa",    desc="Chamber pressure")
nozzle.add("T0",      3500,  "K",     desc="Chamber temperature")
nozzle.add("gamma",   1.2)
nozzle.add("A_throat",0.005, "m^2")
nozzle.add("A_exit",  0.08,  "m^2")
nozzle.add("P_amb",   0,     "Pa")    # vacuum

# Wire in Relations (by object, string name, or inline function)
nozzle.use("nozzle_area_ratio")          # from global registry
nozzle.use("area_mach_supersonic")
nozzle.use("isentropic_ratios")
nozzle.use(my_thrust_fn)                  # inline function

# Update a quantity value
nozzle.set(T0=3200)   # keyword style
python

sys.add(name, value, unit=None, *, desc="")

System.use() signature

sys.use(func_or_relation_or_str, map=None, outputs=None)
→ map={"func_param": "workspace_name"} renames inputs

        map={"P": "P_water", "T": "T_water"},

Optimization

# Maximize / minimize an output by varying design variables
opt = sys.optimize(
    objective="Isp",           # output key to optimize
    design_vars={
        "P0":    (5e6,  30e6),  # (lo, hi) in declared units
        "T0":    (2500, 4000),
    },
    minimize=False,             # True = minimize, False = maximize
    method="differential_evolution",  # or "L-BFGS-B", "SLSQP"
    seed=42, maxiter=500, tol=1e-4,
)

print(opt.fun)        # optimal objective value
print(opt.x)          # dict of optimal design variable values
print(opt.success)    # convergence flag
print(opt["thrust"]) # any workspace quantity at optimum
python
Numerical tip: opt.success may be False if the convergence tolerance wasn't fully met, but the best point found is still valid. Check opt.fun rather than opt.success to assess solution quality.

# Normalized sensitivities: how much does each input drive each output? (∂out/∂in × in/out)

sens.summary(outputs=["thrust"])

for inp, val in sens.top("Isp", n=5):
    print(f"  {inp}: {val:+.4f}")

T0 has 0.5 (square root relationship). Gamma has negative sensitivity — higher gamma → lower Isp/thrust efficiency.

Registry

The registry is the backbone of Anvil's reusability story. It is a SQLite database stored at ~/.anvil/registry.db. Every RSQ (Relation) you push is serialized as its full Python source text and stored with metadata — domain, tags, description, version. On any machine with Anvil installed, you can retrieve and run that RSQ by name without re-defining it.

anvil.push(fn) ──────────────────────────── ▶ ~/.anvil/registry.db │ stores: { name, source_code, domain, │ SQLite table: rsqs tags, description, version } │ │ anvil.R.my_rsq(...) ◀──── deserialize + exec source ◀──┘ lazy: compiled on first call

Self-containment Rule

All imports must be inside the function body. The registry stores only the function's source code — the surrounding module namespace is not captured. When the RSQ is later deserialized and called, np, scipy, and Q are not in scope unless you import them inside the function. This is the most common source of registry bugs.
# CORRECT — imports inside body
def bernoulli(P1, v1, rho, P2):
    import numpy as np
    from anvil import Q
    v2 = np.sqrt(2*(P1 - P2)/rho + v1**2)
    return {"v2": Q(v2, "m/s")}

# WRONG — top-level import won't survive serialization
import numpy as np          # ← not stored in registry
def bernoulli_bad(P1, v1, rho, P2):
    v2 = np.sqrt(...)         # ← NameError when loaded from registry
    return {"v2": v2}
python

How Auto-Seeding Works

On the very first import anvil, Anvil calls seed() which inserts all 76 built-in RSQs into the global registry. On subsequent imports, it compares source strings — if any built-in has changed (e.g., after an Anvil upgrade), only the changed RSQs are re-seeded. Your own pushed RSQs are never touched by seeding.

# Check registry health — verify all RSQs can be loaded
anvil.check()
# Loaded 76 built-in + N user RSQs. All OK.
python

Project Registry (isolated store)

Project registries let you maintain RSQs that are specific to a study or deliverable — without polluting the global registry shared across all projects.

# Create/open a project registry
proj = anvil.project("orbit_study", path="./orbit_work")
# Creates ./orbit_work/orbit_study.db if it doesn't exist

proj.push(my_rsq, domain="orbital")
r = proj.R.my_rsq(x=3, y=4)

# Use as context manager — push() routes to project automatically
with anvil.project("study", path="./work") as p:
    anvil.push(fn1)   # goes to project, not global
    anvil.push(fn2)

# Promote a project RSQ to the global registry
proj.promote("my_rsq")

# Project has same R., S., QDB. namespaces
proj.R.my_rsq(x=3, y=4)
proj.info("my_rsq")
proj.list()
python
Global vs Project: Global registry (anvil.R.*) is shared across all work on the machine. Project registry (proj.R.*) is isolated to a file you control — good for deliverable-specific RSQs, team sharing (just distribute the .db file), or work-in-progress that isn't ready for global promotion.

Adapters

Adapters wrap external Python libraries or CLI tools as native Anvil Relations. Once wrapped, they plug into Systems, sweeps, and sensitivity analysis identically to plain functions.

from anvil import Adapter, Q

def _call(P, T):
    try:
        import CoolProp.CoolProp as CP
    except ImportError:
        raise ImportError("Install: pip install CoolProp")
    return {
        "rho": Q(CP.PropsSI('D', 'P', P, 'T', T, 'Water'), "kg/m^3"),
        "mu":  Q(CP.PropsSI('V', 'P', P, 'T', T, 'Water'), "Pa*s"),
    }

water = Adapter("coolprop_water",
    backend="python", call=_call,
    inputs={
        "P": {"unit": "Pa", "desc": "Pressure", "default": 101325},
        "T": {"unit": "K",  "desc": "Temperature", "default": 300},
    },
    outputs={
        "rho": {"unit": "kg/m^3", "desc": "Density"},
        "mu":  {"unit": "Pa*s",   "desc": "Dynamic viscosity"},
    },
    desc="Water properties via CoolProp",
    tags=["fluid", "water", "coolprop"],
)

# Direct call
r = water(P=101325, T=300)   # rho: 998.2 kg/m^3

# In a System — pass the Adapter object directly (NOT anvil.R.name)
sys.use(water)
python
Registry limitation: anvil.push(adapter) stores the adapter for discoverability but cannot correctly reconstruct the input spec on load. Always pass the Adapter object directly to sys.use() — do not use anvil.R.adapter_name in sys.use().

Adapter Input / Output Spec

FieldKeyDescription
"unit"inputTool expects this unit. Anvil converts SI → tool unit before calling wrapper.
"default"inputDefault value if not in workspace.
"desc"bothHuman description.
"unit"outputWrapper returns in this unit; wrapped as Q automatically.
String inputs: For non-numeric parameters (species names, mode flags), omit "unit". Values pass through unchanged as Python strings/booleans.
Vector inputs: Anvil workspace stores named scalars. Flatten r0=[x,y,z] → r0_x, r0_y, r0_z as separate inputs, then reassemble inside the wrapper.

Standalone solvers in anvil.solvers, also available as System methods.

from anvil import solvers
import numpy as np

# Root finding
sol = solvers.find_root(lambda x: x**3 - x - 1, x0=1.5)
# sol.x = 1.3247  (real root of x^3 - x - 1 = 0)

# Nonlinear system of equations
def equations(vars):
    x, y = vars
    return [x**2 + y**2 - 1, x - y**2]
sol = solvers.solve_nonlinear(equations, x0=[0.7, 0.8])

# ODE initial value problem
def free_fall(t, y):          # y = [h, v]
    h, v = y
    return [v, -9.81]         # dh/dt = v, dv/dt = -g

sol = solvers.solve_ode(free_fall, y0=[1000, 0],
                          t_span=(0, 14), t_eval=np.linspace(0, 14, 100))
# sol.y[0]: height history   sol.y[1]: velocity history

# ODE stiff solver (BDF/Radau)
sol = solvers.solve_ode_stiff(stiff_fn, y0=[...], t_span=(...))

# Global optimization
sol = solvers.minimize_global(lambda x: (x[0]-2)**2 + (x[1]-3)**2,
                                bounds=[(-5,5),(-5,5)],
                                method="differential_evolution", seed=0)

Common Patterns

Pattern 1 — Build a System from Scratch

import anvil
from anvil import Q, System

# 1. Define Relations (or use built-ins)
def reynolds_number(rho, V, L_char, mu):
    from anvil import Q
    return {"Re": Q(rho * V * L_char / mu, "1")}

# 2. Create System and add quantities
pipe = System("pipe_flow")
pipe.add("V",      2.0,   "m/s")
pipe.add("D",      0.05,  "m")
pipe.add("T_water", 300,  "K")
pipe.add("P_water", 1e5,  "Pa")

# 3. Wire in Relations
pipe.use("isa_atmosphere")      # or your own function
pipe.use(reynolds_number, map={"L_char": "D"})

# 4. Solve
result = pipe.solve_forward()
result.summary()
python

Pattern 2 — Chaining RSQs (output feeds next)

# RSQ outputs automatically become available to downstream RSQs
nozzle = System("nozzle")
nozzle.add("M_exit", 3.0);  nozzle.add("gamma", 1.4)
nozzle.add("P0", 5e6, "Pa")
nozzle.add("T0", 3000, "K")

nozzle.use("isentropic_ratios")    # outputs: T0_T, P0_P, rho0_rho
nozzle.use("nozzle_area_ratio")    # uses M_exit, gamma → A_Astar

# Add downstream calculations inline
nozzle.use(lambda T0, T0_T: {"T_exit": Q(T0 / T0_T, "K")})
nozzle.use(lambda P0, P0_P: {"P_exit": Q(P0 / P0_P, "Pa")})
